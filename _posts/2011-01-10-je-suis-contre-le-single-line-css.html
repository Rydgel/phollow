La façon de coder est très personnelle. Ce billet n'a pas pour but de vous forcer à choisir une méthode ou une autre, mais après avoir testé plusieurs méthodes j'en suis arrivé à la conclusion que le single-line CSS n'est pas la meilleure solution.

Dans un article de Simon Collison, celui-ci explique pourquoi il aime coder ces CSS sur une seule ligne. Pour lui cette technique permet de :

  • Faciliter la lecture, et de trouver plus rapidement ce que l'on cherche
  • Réduire la taille du fichier
  • Faire gagner du temps à la création/édition de CSS

J'ai programmé avec les deux méthodes, à savoir le multi-line CSS & le single-line CSS — Et je pense que l'on peut avoir un code CSS maintenable, facilement éditable  et avec une taille réduite sans avoir forcément recours au single-line. Mon contexte de travail est celui d'une équipe d'intégrateurs/designers, mais j'aurais les mêmes pratiques si je devais travailler seul.

Arrête le “scan” de CSS

C'est fatiguant pour les yeux, c'est long et inutile. Pourquoi ne pas rechercher le selecteur avec un mot-clef ? Encore mieux, il suffit d'ouvrir Firebug, de cibler l'élément à éditer pour voir apparaître la ligne du CSS responsable. Ça prend juste quelques secondes et ça vous évite d'avoir à scroller le CSS de haut en bas.

Je range généralement les selecteurs dans l'ordre dans lequel ils apparaissent dans le DOM. Ensuite pour chaque élément je range les propriétés par ordre alphabétique. Comme ça je me doute bien qu'un z-index se trouvera généralement vers le bas et un border-radius en haut.

Le single-line n'est pas propre

Avec l'émergence du CSS3 qui n'est pas encore terminé, mais pourtant utilisable, on se retrouve avec des tas de propriétés constructeurs — vous savez celles qui commence par -webkit-, -moz-, -o- etc. Elles sont généralement plus complexes et donc plus longue, les dégradés par exemple peuvent avoir une tonne d'arguments.

{% highlight css %} div { -moz-border-radius-bottomleft: 4px; -moz-border-radius-bottomright: 5px; -moz-border-radius-topleft: 2px; -moz-border-radius-topright: 3px; -webkit-border-bottom-left-radius: 4px; -webkit-border-bottom-right-radius: 5px; -webkit-border-top-left-radius: 2px; -webkit-border-top-right-radius: 3px; border-bottom-left-radius: 4px; border-bottom-right-radius: 5px; border-top-left-radius: 2px; border-top-right-radius: 3px; } {% endhighlight %} {% highlight css %} div { -moz-border-radius-bottomleft: 4px; -moz-border-radius-bottomright: 5px; -moz-border-radius-topleft: 2px; -moz-border-radius-topright: 3px; -webkit-border-bottom-left-radius: 4px; -webkit-border-bottom-right-radius: 5px; -webkit-border-top-left-radius: 2px; -webkit-border-top-right-radius: 3px; border-bottom-left-radius: 4px; border-bottom-right-radius: 5px; border-top-left-radius: 2px; border-top-right-radius: 3px; } {% endhighlight %}

Je ne sais pas pour vous, mais moi j'ai une haîne profonde du scroll horizontal quand je code.

Les outils de versionning

Le single-line CSS est vraiment désaventageux quand on utilise un outil tel que Git ou Subversion — et si vous n'en utilisez pas, vous devriez sérieusement y songer. Ces outils travaillent ligne par ligne pour montrer les différences entres deux versions du même fichier. Vous voyez où je veux en venir.. on est beaucoup plus précis quand on écrit son CSS sur plusieurs lignes.

Et la taille alors ?

Là non plus il ne faut pas être bête. La taille n'a aucune importance en phase de développement. Et pour la mise en production, il y a des outils pour ça. YUI Compressor et Minify sont deux outils forts pratiques pour ça. Et on peut garder sa version non-compressé pour re-travailler dessus plus tard.

Je ne dénigre pas ceux qui aiment cette méthode, si ça marche bien pour vous, tant mieux. Mais je trouve que lorsqu'on travaille en équipe, c'est plus efficace de leurs faire gagner du temps à la re-lecture.
